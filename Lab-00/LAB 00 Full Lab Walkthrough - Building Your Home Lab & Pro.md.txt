## **PART B: Virtual Network Architecture**

*Estimated Time: 20 minutes*

---

### **B1: Understanding VirtualBox Network Modes**

Before configuring, here's what each mode does and when we use it:

| **Mode** | **Internet Access** | **VM-to-VM** | **Host-to-VM** | **Our Use Case** |
| --- | --- | --- | --- | --- |
| **NAT** | ✅ | ❌ | ❌ | Quick internet for single VM |
| **NAT Network** | ✅ | ✅ | ❌ | VMs sharing internet access |
| **Host-Only** | ❌ | ✅ | ✅ | Simulated office LAN — our primary lab network |
| **Bridged** | ✅ | ✅ | ✅ | Not used (exposes VMs to real network) |
| **Internal** | ❌ | ✅ | ❌ | Isolated testing (future labs) |

**Our architecture uses two networks per VM:**

- **Adapter 1 (NAT):** Gives each VM internet access for updates, downloads
- **Adapter 2 (Host-Only):** Simulates the office LAN where all machines see each other and the host can access them

---

### **B2: Create/Configure Host-Only Network**

1. In VirtualBox Manager, go to **File → Tools → Network Manager**
    
    (Or **File → Host Network Manager** in some versions)
    
2. Click the **Create** button to create a new Host-Only Network
    - It will be named **`VirtualBox Host-Only Ethernet Adapter`** (or **`vboxnet0`** on some versions)
3. Select the new network and click **Properties**
4. Configure the **Adapter** tab:
    
    ```
    text
    IPv4 Address:     192.168.56.1
    IPv4 Network Mask: 255.255.255.0
    ```
    
5. Configure the **DHCP Server** tab:
    - **Enable Server:** ✅ Checked
    
    ```
    text
    Server Address:   192.168.56.100
    Server Mask:      255.255.255.0
    Lower Bound:      192.168.56.100
    Upper Bound:      192.168.56.200
    ```
    
    > ***Why enable DHCP? Client VMs will get addresses automatically during initial setup. We'll assign static IPs later in specific labs. The osTicket server will get a static IP manually.***
    > 
6. Click **Apply** then **Close**

### **D4: Post-Install System Configuration**

1. **Update the system:**
    
    ```
    
    sudo apt update &&sudo apt upgrade -y
    ```
    
    Enter your password when prompted. This may take several minutes.
    
2. **Configure the static IP on the Host-Only adapter:**
    
    First, identify your network interfaces:
    
    ```
    
    ip addr show
    ```
    
    You should see:
    
    - **`lo`** — loopback
    - **`enp0s3`** — NAT adapter (has a 10.0.2.x address)
    - **`enp0s8`** — Host-Only adapter (may have a 192.168.56.x DHCP address or no address)
    
    ![image.png](attachment:258917c1-5915-4a4d-840a-5884ec55b34d:image.png)
    
3. **Edit the Netplan configuration:**
    
    ```
    
    sudo nano /etc/netplan/00-installer-config.yaml
    ```
    
4. **Replace the contents** with (be careful with YAML spacing — use exactly 2 spaces for each indent level, **no tabs**):
    
    ```
    YAML
    network:
      ethernets:
        enp0s3:
          dhcp4:true
        enp0s8:
          addresses:
            -192.168.56.10/24
      version:2
    ```
    
    > ***What this does: ← FIXED
        ◦ `enp0s3` (NAT): Gets IP automatically for internet access
        ◦ `enp0s8` (Host-Only): Gets static IP 192.168.56.10 — this is how your host machine will reach osTicket***
    > 
5. **Save and exit:** Press **`Ctrl+O`**, then **`Enter`**, then **`Ctrl+X`**
6. **Apply the network configuration:**
    
    ```
    
    sudo netplan apply
    ```
    
7. **Verify the static IP:**
    
    ```
    
    ip addr show enp0s8
    ```
    
    You should see:
    
    ```
    
    inet 192.168.56.10/24
    ```
    
8. **Verify internet connectivity:** 
    
    ```
    
    ping -c 4 google.com
    ```
    
    You should see 4 successful replies.
    
    ![image.png](attachment:2b6779e4-b541-4316-859f-523bb14f71f0:image.png)
    
9. **Test connectivity from your HOST machine.** Open Command Prompt on your Alienware: 
    
    ```
    
    ping 192.168.56.10
    ```
    
    You should see successful replies. If not, verify the Host-Only network is configured correctly in Part B.
    
    ![image.png](attachment:140561fd-decd-4692-96b3-4a1df8893669:image.png)

### **D8: Download and Install osTicket**

1. **Install unzip utility:**
    
    ```
    
    sudo apt install unzip -y
    ```
    
2. **Download osTicket** (check [**https://osticket.com/download/**](https://osticket.com/download/) for the latest version — using v1.18.1 as reference):
    
    ```
    
    cd /tmp
    wget https://github.com/osTicket/osTicket/releases/download/v1.18.1/osTicket-v1.18.1.zip
    ```
    
    > ***If this URL doesn't work: Go to `https://github.com/osTicket/osTicket/releases` in your host browser, find the latest release, right-click the .zip download link, copy the URL, and paste it into the wget command.***
    > 
3. **Extract osTicket:**
    
    ```
    
    unzip osTicket-v1.18.1.zip -d /tmp/osticket
    ```
    
4. **Move the upload directory to Apache's web root:**
    
    ```
    
    sudo mv /tmp/osticket/upload /var/www/html/osticket
    ```
    
5. **Create the configuration file from the sample:**
    
    ```
    
    sudo cp /var/www/html/osticket/include/ost-sampleconfig.php /var/www/html/osticket/include/ost-config.php
    ```

### **F1: Why Documentation Matters**

Building this lab is an accomplishment. But if you can't show what
you built and explain what you learned, it's invisible to employers,
peers, and your future self.

The technical skills from this lab — virtualization, networking,
Linux server administration, web application deployment — are
valuable. But the ability to **document what you did, why you did
it, and what the outcome was** is what separates a hobbyist from a
professional.

Every IT department runs on documentation. Every ticket requires
resolution notes. Every project needs a summary. If you build the
habit now, it becomes second nature in the workplace.

---

### **F2: Choose Your Platform**

How you document is less important than **that** you document.
Choose whatever platform fits your workflow:

| Platform | Best For | Cost |
| --- | --- | --- |
| **GitHub / GitLab** | Code-focused portfolios, version control, technical audiences | Free |
| **Hashnode / [Dev.to](http://dev.to/)** | Blog-style write-ups, community engagement, SEO visibility | Free |
| **Medium** | Long-form writing, broader non-technical audience | Free (limited) |
| **WordPress** | Full website control, custom branding | Free (with hosting costs for custom domain) |
| **Notion** | Personal knowledge base, internal documentation | Free |
| **LinkedIn Articles** | Direct employer visibility, professional networking | Free |
| **Physical Notebook** | Personal reference, interview prep | Cost of notebook |

> ***My recommendation: Use at least two — one for detailed technical
documentation (like GitHub or a blog) and one for professional
visibility (like LinkedIn). The detailed version proves depth. The
professional version gets you noticed.***
> 

Whatever you choose, be consistent. A portfolio with 15 documented
labs on any platform is more impressive than a perfectly designed
website with 2 posts.

---

### **F3: Use the STAR Method**

Every lab you complete should be documented using the **STAR method**
— the same framework used in behavioral job interviews. This serves
two purposes: it structures your portfolio AND it prepares you to
talk about your work in interviews.

**STAR stands for:**

```
S — SITUATION
    What was the context? Who needed help? What was the
    business impact? What made it urgent?

    Example: "NovaBridge Technologies, a managed service
    provider, signed three new clients. Field technicians
    needed standardized bootable USB toolkits before
    onboarding site visits the following week."

T — TASK
    What were YOU specifically responsible for? What were
    the constraints — deadline, tools available, scope?

    Example: "As the IT Support Specialist, I needed to
    create three types of bootable media and write
    documentation so any technician could recreate
    the toolkit independently."

A — ACTION
    What did you actually DO? Be specific — name the tools,
    the commands, the steps. Include what DIDN'T work and
    how you pivoted. This is where your technical depth shows.

    Example: "I used Rufus to create a Windows 11 GPT
    bootable USB, Ventoy for a multi-boot Linux diagnostic
    drive, and the built-in Windows Recovery Drive tool.
    During testing, the Windows USB failed to boot on a
    UEFI machine. I diagnosed the root cause using diskpart,
    identified an MBR/GPT mismatch, and recreated the USB
    with the correct partition scheme."

R — RESULT
    What was the measurable outcome? How many users were
    impacted? How much time was saved? What did you prevent
    from happening again?

    Example: "All three toolkits were delivered 2 days before
    deadline. The Ventoy USB consolidated 4 tools into one
    drive, reducing equipment per technician by 75%. The
    documentation I created was adopted as the standard
    procedure, preventing future boot failures."
```

### **STAR Template**

Use this template for every lab you complete:

# STAR Writeup — Lab [XX]: [TITLE]

## Situation

[WHO] at [COMPANY] experienced [PROBLEM] which caused
[BUSINESS IMPACT]. [URGENCY/CONTEXT details].

## Task

As [YOUR ROLE], I needed to [OBJECTIVE] while ensuring
[CONSTRAINT]. The deadline was [TIMEFRAME] and
[ADDITIONAL REQUIREMENTS].

## Action

1. **Initial Assessment:** [What I checked first and
what I observed]
2. **Diagnosis:** [What tools/commands I used and what
I discovered]
3. **Resolution:** [Specific technical steps I took to
fix the issue]
4. **Verification:** [How I confirmed the fix worked]
5. **Prevention:** [What I documented or recommended to
prevent recurrence]

### What Didn't Work (and Why)

[Describe the first approach that failed and what it
taught you about the root cause — this shows
troubleshooting depth]

## Result

- ✅ [Primary outcome with specific metric]
- ✅ [Secondary outcome — users impacted, time saved]
- ✅ [Preventive measure implemented]
- 📝 Documented in ticketing system with full root
cause analysis

> ***Why include "What Didn't Work"? Because troubleshootingis never a straight line. Showing your first attempt faileddemonstrates that you can adapt, learn, and persist.Interviewers specifically look for this. Anyone can describea success — professionals can describe the failures that ledto the success.***
> 

---

### **F4: What to Document for Each Lab**

At minimum, every lab should produce:

1. WHAT YOU BUILT / FIXED
    - Screenshots or screen recordings of key moments
    - Before and after states
    - The tools and commands you used
2. WHAT YOU LEARNED
    - Key concepts (in your own words, not textbook definitions)
    - What surprised you
    - What you'd do differently next time
3. HOW IT CONNECTS TO THE JOB
    - Which IT roles use this skill daily
    - How you'd explain this to a non-technical manager

    - If you didn't: write resolution notes anyway in whatever
    format you choose
    - The habit of documenting resolutions is more valuable
    than the platform you use

> ***The goal is not perfection. The goal is consistency. A roughwrite-up published every week beats a polished masterpiecepublished never. Start messy. Get better as you go. The firstpost is always the hardest — every one after that gets easier.***
